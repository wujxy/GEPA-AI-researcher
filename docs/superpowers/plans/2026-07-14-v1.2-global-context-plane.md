# GEPA v1.2 Global Context Plane Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the v1.2 Global Context Plane so agent prompts are generated from traceable context views instead of temporary `_gepa_context` dictionaries.

**Architecture:** Keep the current Proposer / Executor / Judger roles, but insert a formal context layer between stores and prompts. Existing stores remain the source of truth; `GlobalContextPlane` reads them, emits `ContextBlock` objects, `ContextViewBuilder` selects role-specific views, and `PromptAssembler` renders bounded prompts.

**Tech Stack:** Python dataclasses, JSON/JSONL stores, existing GEPA domain/store patterns, pytest.

## Global Constraints

- Do not add Planner or Critic in v1.2.
- Do not introduce embedding, vector database, or compact agent in v1.2.
- Context Plane stores complete local facts; prompt budgeting only affects PromptAssembler rendering.
- `_gepa_context` must stop being the primary context transport once migration is complete.
- Context provenance is mandatory for blocks derived from candidates, executions, events, artifacts, or files.
- File cache keys must include repository identity, commit sha, path, and content hash.
- Role views must be deterministic for the same run state.
- Existing user config canonical entry remains unchanged.

Selection rules:

- Proposer view includes:
  - run facts
  - loop state
  - selected parent candidate facts
  - score summaries from candidate cards
  - recent judge feedback if available as blocks
- Executor view includes:
  - run facts
  - current candidate fact
  - workspace metadata
  - evaluation phase/sample ids

- Proposer view must not include full raw execution stdout/stderr by default.
- Executor view must not include unrelated parallel candidate facts.
